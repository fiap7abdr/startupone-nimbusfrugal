# i18n — Design Spec

## Summary

Add internationalization (pt-BR and en) to the entire Nimbus Frugal application using next-intl. Locale is resolved automatically from the browser and persisted in a cookie. A manual switcher lets the user override. No URL prefix changes.

## Languages

- `pt-BR` (default, fallback)
- `en`

## Library

**next-intl** — designed for Next.js App Router with native Server Component and Client Component support.

## Locale Resolution

Order of precedence:

1. Cookie `NEXT_LOCALE` (user explicitly chose)
2. `Accept-Language` header (browser detection)
3. Fallback: `pt-BR`

## Middleware

The existing `middleware.ts` handles auth (cookie-based session check). It will be extended to also resolve the locale:

- Read `NEXT_LOCALE` cookie or parse `Accept-Language` header
- Set `NEXT_LOCALE` cookie if not present (so subsequent requests are consistent)
- Pass locale to next-intl via `createMiddleware` or manual composition

Constraint: middleware must stay under 1MB (Edge, Vercel free tier). next-intl's middleware is lightweight and does not import heavy deps.

## next-intl Setup

### Configuration file

`i18n/request.ts` — exports `getRequestConfig` that reads the locale from the request and loads the corresponding JSON.

### Server Components

Use `getTranslations(namespace)` to get a `t()` function:

```tsx
import { getTranslations } from "next-intl/server";

export default async function DashboardPage() {
  const t = await getTranslations("dashboard");
  return <h1>{t("title")}</h1>;
}
```

### Client Components

Wrap the app in `NextIntlClientProvider` (in root or app layout) passing `messages` and `locale`. Client Components use `useTranslations(namespace)`:

```tsx
"use client";
import { useTranslations } from "next-intl";

export function Sidebar() {
  const t = useTranslations("sidebar");
  return <span>{t("dashboard")}</span>;
}
```

### Provider placement

`app/layout.tsx` (root layout) wraps children with `NextIntlClientProvider`. The locale and messages are resolved server-side and passed as props.

## Locale Switcher

A `LocaleSwitcher` component — simple dropdown with two options (PT / EN).

**Placement:**
- Marketing pages: in the `SiteHeader`
- App pages: in the sidebar footer (above logout)
- Admin pages: in the admin sidebar footer

**Behavior:**
- Sets `NEXT_LOCALE` cookie via a server action
- Calls `router.refresh()` to re-render the page with the new locale

## Zod Validation Messages

Zod error messages in `lib/validations.ts` are currently hardcoded in Portuguese. They will be replaced with i18n keys using Zod's `message` option, resolved at the point of use (server action) using `getTranslations("validation")`.

## Scope

### What gets translated

- All 25 page files (marketing, auth, app, admin, setup)
- All 20 component files with user-facing text
- Sidebar navigation labels
- Button labels, badges, empty states, error messages
- Modal titles and descriptions
- Zod validation error messages
- Page metadata (titles, descriptions)

### What does NOT get translated

- Database content (org names, tenant names, account names, connector types)
- Connector type identifiers (aws_organizations, cur, etc.) — only their display labels via `connectorLabel()` get translated
- CSS/design tokens
- URLs (no locale prefix)
- Admin-only technical labels that are universal (ARN, External ID, etc.)
